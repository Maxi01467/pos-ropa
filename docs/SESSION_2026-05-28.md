# Registro de cambios - 2026-05-28

Este archivo resume todo lo que se modificó durante esta sesión para investigar y mitigar el error 500 en la versión web.

## Cambios realizados

- Se ajustó `src/components/layout/route-guard.tsx` para que `pathname` sea tratado de forma segura cuando `usePathname()` todavía no devuelve un valor estable.
- Se ajustó `src/components/layout/pos-layout-client.tsx` para evitar llamadas directas a helpers de ruta cuando `pathname` es nulo o todavía no está listo.
- Se reforzó `src/lib/offline-config.ts` para que PowerSync solo se habilite si existen credenciales completas de servidor.
- Se endureció `src/app/api/powersync/auth/route.ts` para aceptar tanto clave privada en PEM como en base64, y para responder `503` si la configuración no está lista.
- Se cambió `src/components/layout/powersync-provider.tsx` para que PowerSync solo arranque en runtime de escritorio y no en la web normal.
- Se cambió `src/lib/offline/offline-bootstrap.ts` para que el bootstrap offline no inicialice PowerSync en una sesión web común.

## Validaciones ejecutadas

- Se ejecutó `npm run build` varias veces y compiló correctamente.
- Se verificó en navegador local que la home y el login cargan sin error de servidor.
- Se confirmó que, en la web normal, ya no se dispara la inicialización de PowerSync al entrar cuando el build nuevo está activo.

## Observaciones

- El cambio grande que introdujo el comportamiento nuevo fue el push `Mejoras de diseño y animaciones en nueva venta y caja`.
- La investigación apuntó primero a rutas y guards de cliente, pero el foco real terminó estando en el arranque de PowerSync y el bootstrap offline global.
- Si el 500 reaparece en producción, el siguiente paso es revisar el entorno de despliegue y las variables reales del host.
